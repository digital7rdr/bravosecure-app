   Bravo Secure — Messenger Module Bug Audit

**Date:** 2026-06-26  |  **Branch:** `fix/batch-plan-safe-wins`  |  **Method:** 9-slice parallel code audit + synthesis (report-only, no fixes applied)

> Scope: 1:1 crypto, group crypto, transport/WS/relay/outbox, messages/store, media/vault, group calls (SFU), 1:1 calls, dept-chat/channels/Ops-Room, runtime-lifecycle/backend/push. Bugs the current code already handles are excluded.

## Overall health

The messenger's cryptographic CORE is genuinely robust: the hard, scary parts (1:1 Double Ratchet, sealed-sender v3 cert-in-AAD pre-decrypt verify, session-wipe protection, atomic receive txns, durable 1:1 outbox, reconnect/supersession/JWT-drift survival, encrypt-then-MAC AES media, no plaintext/key logging) are well-hardened and the prior P0s are closed. Robustness drops sharply at three seams. (1) The push-wake pipeline is half-migrated to opaque {eventId,eventClass} and is currently DEAD end-to-end — SOS panic alerts and every server notification silently fail to wake backgrounded/killed devices (one CRITICAL, one HIGH, both reproducible today and verified against the code). (2) Group key delivery/recovery is structurally OWNER-bound, and 'server says member, crypto says keyless' divergence pervades Ops Rooms, dept channels, and same-epoch forks with no reliable automatic recovery. (3) Several freshness/timestamp windows (15-min AAD, 7-day stash prune, send-time disappearing) contradict the documented 30-day relay dwell and silently drop legitimately-delayed messages while the sender shows 'sent'. 1:1 calls have also regressed behind the already-fixed group-call code on three small parity items (focus auto-unmute, camera release, ring-time capture). Net: solid 1:1 text and crypto; fragile push, group-recovery, offline-delivery, and 1:1-call lifecycle — and the highest-leverage fixes are backend/architecture-shaped, not crypto-primitive changes.

**Tally:** 12 active (reproducible today) · 22 latent (edge-case) · 18 future-risk.

### A1. SOS-WAKE-DROPPED-AT-RELAY  — 🔴 Critical

- **Area:** runtime/push (auth-service)
- **Symptom:** When a client or CPO raises SOS, crewed CPOs / the principal with the app backgrounded or killed receive NO push — the panic alert never reaches the device. VERIFIED: both legacy publishers emit a frame with no eventId; the push subscriber bails on !frame.eventId, dropping it before FCM. The correct opaque BookingPushBridge.sosAlert() exists and is unit-tested but is never called from these two production paths.
- **Location:** `apps/auth-service/src/sos/sos.service.ts:124 (and apps/auth-service/src/agents/agent.service.ts:1167)`
- **Fixability:** needs-backend

### A2. OPAQUE-WAKE-NO-CLIENT-CONSUMER  — 🟠 High

- **Area:** runtime/push (mobile + backend)
- **Symptom:** ALL server-driven push wakes (booking-approved, mission-dispatched/aborted, payout-settled, agent-approved/rejected, dispatch-offer, provider-accepted, no-provider, agency-no-show, incident, and SOS) never surface on a backgrounded/killed device. VERIFIED: backend sends data-only {eventId,eventClass} expecting GET /events/by-id/:eventId hydration, but that route exists ONLY in comments (no controller), and the client handlers still key off the now-absent data.kind/bookingId/missionId, so every frame falls through to 'unknown background wake'. Compounds the SOS bug — even a correctly-bridged SOS would not surface.
- **Location:** `src/modules/messenger/push/fcmBootstrap.ts:916`
- **Fixability:** needs-backend

### A3. aad-freshness-drops-offline-messages  — 🟠 High

- **Area:** 1:1 crypto
- **Symptom:** A 1:1 message delivered >15 min after it was sealed is silently dropped by the recipient (sender still shows 'sent'). Hits any recipient who was offline / Doze-frozen / FCM-throttled and returned later, AND every durable-outbox retry that lands >15 min after send. The fixed 15-min SEALED_AAD_SKEW directly contradicts the 30-day relay store-and-forward dwell; aad.ts is stamped once at seal and never re-stamped on redelivery. User even sees a misleading 'a device clock looks wrong' banner.
- **Location:** `src/modules/messenger/runtime/productionRuntime.ts:4799`
- **Fixability:** needs-arch-signoff

### A4. RC3-group-fanout-outbox-gap  — 🟠 High

- **Area:** transport / group send
- **Symptom:** When a group member's session can't be established at send time (unprovisioned, exhausted OPK pool, keys-service blip), that member silently never receives the message and there is NO durable retry — establish-session + encrypt run BEFORE sqlOutbox.enqueue, so a throw leaves no row to replay. The sender's bubble still flips to 'sent' as long as one other peer succeeded, making the loss invisible.
- **Location:** `src/modules/messenger/runtime/productionRuntime.ts:1739`
- **Fixability:** fix-now-medium-risk

### A5. focus-gain-no-unmute  — 🟠 High

- **Area:** 1:1 calls
- **Symptom:** A transient audio-focus loss during a 1:1 call (incoming GSM call, alarm, another app's sound) auto-mutes the caller; when focus returns the 'Paused' banner clears but the mic stays muted, so the user is inaudible to the peer with no signal until they manually unmute. The GAIN branch only clears the banner and never re-toggles mute; GroupCallScreen's BS-FOCUS-UNMUTE does restore it — the 1:1 path omitted the unmute half.
- **Location:** `src/screens/messenger/CallScreen.tsx:238`
- **Fixability:** fix-now-low-risk

### A6. opk-refill-overwrites-live-prekeys  — 🟡 Medium

- **Area:** 1:1 crypto
- **Symptom:** After a chat-heavy session triggers an own-OPK self-refill (poolSize<10), still-valid one-time prekeys get bricked: the refill probes from keyId=1 for the FIRST empty slot then fills 50 contiguous ids over still-occupied higher keyIds, overwriting their private halves while the server keeps the old public halves. Any cold-contact sender who popped one of those ids gets a PreKeyWhisperMessage that fails X3DH (Bad MAC), losing that contact's first message until rehandshake. Fix: fill from max(keyId)+1, not the first gap.
- **Location:** `src/modules/messenger/runtime/productionRuntime.ts:3478`
- **Fixability:** fix-now-medium-risk

### A7. GROUP-REACT-AUTHOR-BLIND  — 🟡 Medium

- **Area:** messages/store (group)
- **Symptom:** In a group chat the SENDER of a message never sees reactions others place on it, and reply jump-to-original on their own messages misses. Root cause: the group send mints msgId for the local bubble id but a DIFFERENT clientMsgId on the wire; every recipient stores under clientMsgId, so reactors target clientMsgId while the author's local id is msgId — applyReaction's id match misses on the author's device. The 1:1 path is immune (sets clientMsgId=msgId).
- **Location:** `src/modules/messenger/runtime/productionRuntime.ts:1699`
- **Fixability:** fix-now-low-risk

The group-call path was fixed to stop the track; the 1:1 parity fix was never applied.

### A9. incoming-camera-before-accept  — 🟡 Medium

- **Area:** 1:1 calls
- **Symptom:** When a 1:1 call rings with the app in the foreground, the mic and (for video) the camera are acquired immediately, before the user taps Accept — the privacy LED lights and a self-preview shows during ring, and declining still briefly activated the camera/mic. Media is captured at mount for direction==='incoming' rather than deferred to controller.accept().
- **Location:** `src/modules/messenger/webrtc/useCall.ts:363`
- **Fixability:** fix-now-medium-risk

### A10. r2-media-never-purged  — 🟡 Medium

- **Area:** media
- **Symptom:** Encrypted attachment ciphertext for disappearing (TTL) and retracted messages persists in R2 indefinitely — messenger-service has no DeleteObject anywhere; retract/expire only purge the local SQLCipher row, local blob cache, and relay envelope. Combined with the lax-grant/30-day gap, vanished media is still re-downloadable with the in-band key. The P0-A4 envelope->object linkage is referenced client-side but unimplemented server-side.
- **Location:** `apps/messenger-service/src/media/media.service.ts:178`
- **Fixability:** needs-backend

### A11. self-1to1-media-mistyped-file  — ⚪ Low

VERIFIED: the 1:1 local-echo bubble hardcodes type: opts.attachment ? 'file' : 'text' instead of attachmentMessageType(...). Regresses the just-shipped tap-to-record voice note on the 1:1 sender side.

## 3. Future risks — will bite as the app scales / changes

- **F1. transcript-fork-detection-inert** (group crypto) — transcriptHash is computed and chained locally but NEVER compared (the wire GroupAdminAction carries no peer hash and no receiver reads it). The documented P1-G1 protection against a server/admin forking membership (Alice {add Bob} vs Carol {add Eve}) is dead — it bites the moment a malicious or buggy relay/admin forks the membership transcript.
- **F2. planAddAndRekey-nondeterministic-key-fork** (group crypto) — The add planner still mints a RANDOM key while the sibling remove/leave planners were made deterministic (P0-G3). Not reachable single-owner today, but the instant a second admin or a promotion path exists, two admins racing an add at the same base epoch converge on different master keys — the seed of the unrecoverable same-epoch G1 fork.
- **F3. readonly-viewer-enforcement-client-only** (dept-chat / channels) — Viewers of standard/read_only channels are seeded as real Signal group members holding the master key; the only gate on posting is the hidden UI composer — the relay applies no per-message role ACL. A determined viewer can post into any board/announcements channel; the 'managers post, CPOs read' guarantee is cosmetic.
- **F4. outbox-retry-reconnect-only** (transport) — drainOutbox runs only at startup and on WS reconnect — there is no retry timer. A message whose relay.send transiently failed (500/timeout) while the WS stays connected can stall for hours on a stable long-lived connection until the next reconnect that lands after next_retry_at.
- **F5. auth-refresh-reopen-skips-listener-teardown** (transport) — After a mid-session token refresh, open() does not removeAllListeners()/null the prior socket; with socket.io forceNew:false + multiplex reuse, the reopened socket can carry a duplicate listener set, dispatching every server frame twice (duplicate decrypt + duplicate state transitions). Bites under frequent refresh + Manager reuse.
- **F6. minimize-leaks-sfu-handler** (group calls (SFU)) — Each minimize->restore cycle leaks the prior boot's SFU frame handler (the keepAlive early-return skips cleanupSubRef). Handlers accumulate; a peer who joins or switches audio->video after a restore can be double-consumed or routed into the unmounted tree, so their tile may never appear.
- **F7. roomtoken-expires-midcall-rejoin** (group calls (SFU)) — A group call longer than the 30-min room-token TTL that then hits a WS reconnect fails to rejoin (room_token_invalid -> 'failed') — attemptSfuRejoin re-sends the original token and there is no in-call token re-mint path.

### DEPT-CHAT / CHANNELS / MISSION OPS ROOM / DISPATCH INTENTS

### Runtime lifecycle / store / foreground-service / backend / push

The runtime rebuild path (productionRuntime.disposeLiveRuntime + authStore.signOut) is mostly solid — heartbeat, sweeper, AppState listener, SFU handles, call dispatchers, and the call foreground-service are all torn down on logout, and the owner-epoch guard correctly fences cross-user frame/state bleed. The FCM cleartext-leak fix is real: the push:events subscriber and BookingPushBridge both forward only opaque {eventId, eventClass}. However, that opacity hardening was shipped WITHOUT its other half: the client never learned to consume {eventId, eventClass} (no /events/by-id route exists anywhere) so every server push wake silently dies on the device, and two SOS publishers still emit the deleted legacy frame shape that the subscriber now drops before FCM — a safety-critical regression.

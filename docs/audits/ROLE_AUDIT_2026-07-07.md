# Role Selection & Role Lifecycle Audit — 2026-07-07

**Scope:** every role system in the platform — selection at registration, assignment, update/downgrade, propagation (JWT / mobile / ops-console), enforcement, and auditability — across the mobile app (`src/`), auth-service (`apps/auth-service`), ops-console (`apps/ops-console`), messenger (`packages/messenger-core`, conversations/department services) and the staging Postgres (Supabase `qkkfkicgoncxslbwhyhz`).

**Method:** DB ground-truth queries (constraints, value distributions, cross-table consistency, audit-log mining) + three full code sweeps (mobile, backend, console/messenger) with file:line evidence.

**Overall score: 59/100.** Role _selection_ and _upgrade_ paths are industry-grade (server-forced role at registration, fresh-DB-read enforcement everywhere). Role _downgrade/revocation_ is where it breaks: terminated agents keep their role forever, removed CPOs keep API access, no role change revokes a session, and no role event is audited. "Is the role perfectly updating?" — **No. Role updates are one-way and sticky.**

**Tier follow-up (lite → pro → no-pay → lite?):** the loop exists and the DB flip works — hourly `ProLapseCron` sweep + Stripe webhooks with a proper grace window — but it leaks at the edges: one of the three live Pro users can **never** be downgraded (NULL expiry exempts them from the sweep, RS-17), auto-renew users depend on a single webhook arriving (RS-18), and the downgrade is **cosmetic in practice** because no endpoint enforces tier (RS-03) and the warm app never notices the flip (RS-19).

## 1. Role taxonomy (what actually exists)

Five independent role stores, plus three chat-level role columns. No single reconciliation point; `resolveAccountKind` (`apps/auth-service/src/auth/account-kind.ts:79-108`) is the de-facto arbiter.

| Store                             | Values (constraint)                                                                 | Values in data                                                         | Purpose                                |
| --------------------------------- | ----------------------------------------------------------------------------------- | ---------------------------------------------------------------------- | -------------------------------------- |
| `users.role`                      | CHECK: `individual, corporate, agent, service_provider, ops` (default `individual`) | individual:84, agent:34, service_provider:8, **ops:1**, corporate:0    | platform identity                      |
| `admin_users.role`                | enum `admin_role`: `OPS, SUPERVISOR, ADMIN` (default OPS)                           | ADMIN:1, SUPERVISOR:8, OPS:0                                           | ops-console RBAC                       |
| `agents.type` + `agents.status`   | enum: `company, cpo, transport` × FSM `DRAFT…ACTIVE, REJECTED`                      | providers = `company`                                                  | partner tier/state                     |
| `org_members.member_role`         | CHECK: `cpo, manager` (default cpo)                                                 | cpo:7, **manager:0**                                                   | org workspace authz                    |
| `cpo_pool.role`                   | **free text, no constraint**                                                        | `'cpo'`:4, `'CPO'`:28, `'Senior CPO · Armed'`:3, 7 more label variants | display label misused as a role column |
| `conversation_members.role`       | text                                                                                | admin:34, member:32                                                    | E2E group admin                        |

Mobile adds two more client-side vocabularies: `account_kind` (`individual|agency|cpo`, server-derived) and the `RoleBadge` UI enum (`client|agency|cpo|lead`). Mapping between vocabularies is done ad hoc per file (e.g. `src/screens/agent/agentFlowHelpers.ts:114-116`), never centrally.

**Convention decoded:** a service provider is _also_ a `company`-type agent — all 7 "users with an `agents` row but `role≠agent`" are `service_provider` + `agents.type='company'`. Not a bug, but every consumer must branch on `agents.type`, not `users.role`, or it miscounts.

## 3. Findings

### P1 — fix soon

**RS-01 — Removed/suspended managed CPO keeps full `/agents/*` API access.**
`CpoSessionGuard` (`apps/auth-service/src/auth/cpo-session.guard.ts`) — whose entire documented purpose is mid-session ejection of a non-active CPO — is **mounted on zero routes** (no `@UseGuards` reference anywhere). `OrgCpoService.setMemberStatus` (`org-cpo.service.ts:425-445`) flips `org_members.status` and triggers chat rekey but **does not revoke JTIs**. The user row itself isn't suspended, so `/auth/refresh` keeps issuing tokens indefinitely. Net: a CPO an org has suspended/removed can keep toggling duty, applying to jobs and driving the mission FSM (`agent.controller.ts:35` — `JwtAuthGuard` only) until they happen to log out. Org-scoped and dept-chat routes are safe (fresh `status='active'` reads).
_Fix shape:_ mount `CpoSessionGuard` on the agent surface (or fold the check into `JwtAuthGuard` for `account_kind='cpo'`), and revoke device JTIs in `setMemberStatus`.

**RS-02 — Conversation `removeMember` has no rekey seam (E2E forward secrecy).**
`conversations.service.ts:179-213` deletes the `conversation_members` row with **no membership intent and no rekey**, so a removed member of a mission/DM group keeps the group master key indefinitely. Department channels do this correctly (`department.service.ts:435-450` enqueues remove+rekey intents; messenger-core planners `groupClient.ts:767-972` are sound). ⚠️ Fix touches group-key handling — **architecture stop-condition, needs sign-off before code**.

**RS-03 — Server-side Pro paywall is dead code.**
`TierGuard` / `@RequireTier` (`tier.guard.ts`) is applied to **no handler** — every reference is a comment. Tier enforcement is client-side only (`onTierInsufficient` paywall routing), so any Lite user with a script can call every Pro endpoint. Monetization bypass, not data exposure.

**RS-04 — `agent.terminate` never reverts `users.role` (the "sticky role" defect).**
`OpsService.rejectAgent`/terminate mutate `agents.status` only; no code path anywhere reverts `users.role` (2 live cases above). Consequences: the JWT keeps `role='agent'`; mobile `resolveAuthedRoute` (`src/navigation/resolveRoute.ts:54`) keeps the **agency shell** on the legacy-role branch; `callRoleGate.ts:25` keeps treating them as an agent; ops-console Users counts them as agents. Role goes up, never comes down.

### P2

**RS-05 — No session/JTI revocation on any role flip + messenger ticket re-emits stale claims.**
`UPDATE users SET role='service_provider'` (`agent.service.ts:200-203`) leaves the old token live (≤15 min stale `role` claim). Internally harmless (guards read DB), but the **messenger ticket** (`auth.controller.ts:279`) is minted from the _presented token's_ claims, not a fresh read — stale role/suspension flows into messenger-service for up to 15m + 5m ticket TTL.

**RS-06 — Mobile never refreshes role for client/agency shells.**
`/auth/me` is re-pulled only on cold boot, login, biometric sign-in, or CPO-shell resume (`CpoNavigator.tsx:65-71`). An individual or agency user whose role/kind changes server-side keeps the wrong shell **indefinitely** while the app stays warm. The CPO shell is the only one with an AppState-resume recheck (`authStore.recheckMembership`, authStore.ts:610-630) — and it only acts on the revocation direction.

**RS-07 — `pendingProvider` flag can lock a user in the agency shell against server truth.**
Persisted AsyncStorage flag (`auth:pending_provider`) makes `resolveAuthedRoute` return `'agency'` (`resolveRoute.ts:56`) before the server ever flips the role. It's cleared only on `POST /agents` success/already-exists (`AgentTypeSelectScreen.tsx:94,101`) — **not on failure or logout**. A user who abandons provider signup is stuck in the agency shell across restarts with `users.role='individual'`.

**RS-08 — Last-admin auto-promotion is a bare DB flip outside the E2E transcript.**
`conversations.service.ts:191-212` promotes the oldest member to `role='admin'` with no signed group-admin action. Ops derives E2E admin authority from this roster field (`groupClientAdapter.ts:77-83` → `applyAdminAction` gates, `groupClient.ts:483-486`), so a member silently gains add/remove/rekey power with no cryptographic provenance.

**RS-09 — Admin role management is entirely out-of-band and the invite flow is vaporware.**
No endpoint assigns or changes `admin_users.role` (the only writer is the hard-403'd `adminRegisterVerify`, whose full self-grant-ADMIN logic is still present as dead code — `auth.service.ts:243-340`, defused only by `auth.controller.ts:152-157`). Login page promises an invite flow that doesn't exist (`login/page.tsx:79-81`). Admin roles are set by raw SQL/seed with **zero in-app audit trail**. Also: current data shows 8/9 admins are SUPERVISOR, 0 are OPS — least-privilege isn't being exercised.

**RS-10 — No org role-change path (CPO ⇄ manager), and the seams assume there never will be.**
`member_role` is written once at creation (`org-cpo.service.ts:114,206-211`); no endpoint updates it (only `status`). Channel seeding branches on it (managers → channel `admin` + restricted channels; CPOs → `viewer` + open channels). A DB-level promotion/demotion would _not_ reseed channels or rekey — silent access-control gap if the path is ever added naively. Related: any org manager can mint another `manager` via `POST /org/cpos` with no ops approval and no audit action.

**RS-11 — Role events have no audit trail at all.**
`ops_audit` contains `agent.approve`/`agent.terminate` but **no action for any role change** (user, admin, or org member). `org_audit_log` has no `member.*` actions whatsoever (only incident/attendance/channel). For a security product this is a compliance gap: who became a manager, when an admin was promoted, when a role flipped — unanswerable from the data.

### P2 — subscription-tier lifecycle (lite → pro → lapse, follow-up check)

The healthy part first: `POST /subscription/pro` debits 2000 BC and extends `pro_active_until` +30d in ONE transaction (`subscription.service.ts:52-111`); Stripe auto-renew extends on `invoice.paid`, `invoice.payment_failed` only marks `past_due` (grace until period end), and `customer.subscription.deleted` downgrades once the paid period lapses (`:177-235`). The backstop `ProLapseCron` runs hourly in-process (`pro-lapse.cron.ts:35`, plain setInterval, `PRO_LAPSE_SWEEP_DISABLED` env kill-switch) calling `sweepLapsedPro` (`:243-256`). Data today: 3 Pro users — 2 with valid future expiries (one, `604c77bb`, expires 2026-07-08 and will exercise the sweep live), 0 expired-but-still-pro. The leaks:

**RS-17 — A Pro user with `pro_active_until = NULL` is immortal-Pro.**
`sweepLapsedPro` requires `pro_active_until IS NOT NULL` (`subscription.service.ts:249`), so a NULL-expiry Pro row is never downgraded. 1 of the 3 live Pro users is exactly this (`952291de`, Pro since 2026-05-10, `pro_active_until`/`pro_renew_status`/`stripe_subscription_id` all NULL — a manual grant predating the subscription flow; `subscribeToPro` always sets an expiry, so every NULL is a hand-edit the backstop silently exempts). Worse, the NULL semantics are inconsistent: the `customer.subscription.deleted` webhook treats NULL as _lapsed_ (downgrades, `:226-228`) while the sweep treats NULL as _skip forever_.

**RS-18 — The sweep excludes Stripe-linked rows; a missed webhook = Pro forever.**
`sweepLapsedPro` also requires `stripe_subscription_id IS NULL` (`:248`), delegating every auto-renew user entirely to the `customer.subscription.deleted` webhook. If that single webhook is missed (endpoint down, secret rotated, replay window passed), nothing else ever downgrades them — `invoice.payment_failed` only sets `past_due`. There is no periodic reconciliation against Stripe.

**RS-19 — The lite downgrade is cosmetic: nothing enforces it and the running app never notices.**
Even when the sweep flips the column correctly: (a) the **only** emitter of `403 tier_insufficient` in the backend is the unmounted `TierGuard` (grep confirms no inline throw anywhere) — so the mobile's entire `tier_insufficient` → ProPaywall pipeline (`src/services/api.ts:105-110`, `MainNavigator.tsx:217+`) is dead code today; (b) the warm app never re-pulls `/auth/me` (RS-06) and never compares `pro_active_until` to the clock locally, so a downgraded user keeps the full Pro UI until cold boot. Direct answer to "if he doesn't pay, does he go back to lite?" — **the database column does; the product experience doesn't.**

### P3

**RS-12 — Taxonomy drift: legacy values live in constraints, types, and one real row.**
`users_role_check` still allows `corporate` (0 rows; corporate tables were dropped 2026-07-07) and `ops` (1 legacy row: "Ops-1", 2026-04-25). Meanwhile 8/9 console admins have `users.role='individual'` — the `role='ops'` convention was abandoned after the first admin but never settled or cleaned. Mobile `UserRole` still includes both dead values; `resolveAuthedRoute` has **no branch for either**, so a `corporate` or `ops` user falls through to the client shell (`src/screens/ops/*` exists but is unreachable via the role switch).

**RS-13 — `cpo_pool.role` is an unconstrained display label, case-inconsistent, 36% orphaned.**
Values range over `'cpo'`/`'CPO'`/`'Senior CPO · Armed'` etc.; any `role === 'CPO'` comparison silently misses the 4 lowercase rows. Only 28 of 44 `cpo_pool.id`s match a `users.id` — 16 roster-only phantom rows.

**RS-14 — Staging data hygiene.**
3 `users.role='agent'` accounts named "E2E CPO Agent" have no `agents` row (test fixtures). One historical terminate deleted the `agents` row outright while newer ones set REJECTED — two different terminate semantics in the data.

**RS-15 — Console cosmetics & UX gating.**
Users _list_ relabels `individual`→"Client" but user _detail_ (`users/[id]/page.tsx:174`) and finance (`finance/page.tsx:455`) render the raw DB string. `useOpsMe` caches the admin role in SWR (`revalidateOnFocus:false`) so a role change shows stale buttons until revalidation (server 403s still hold — UX-only). `DisputesTab` gates Resolve with `canAdjustWallet` instead of its own capability (`finance/page.tsx:485,508`). Mobile `DashboardScreen.tsx:586` renders the raw role string in the badge.

**RS-16 — `forbidNonWhitelisted` off by default (`main.ts:51-54`).**
Extra body fields are silently stripped rather than rejected. Not exploitable today (no DTO declares `role`), but it removes the tripwire that would catch a future DTO accidentally exposing one.

## 4. What is solid (keep as-is)

- **No self-selected roles, anywhere.** Registration DTOs have no role field on either side (client `api.ts:204-214` P0-V1; server `register.dto.ts`); server hardcodes `individual`/`lite` (`auth.service.ts:220-228`). The mobile "Service Provider" choice at `RoleSelectionScreen` is only a routing hint — the actual flip happens post-auth via `POST /agents` with a `company`-only whitelist.
- **Enforcement never trusts the JWT role claim.** Every guard re-reads the relevant store per request: `AdminGuard` (+`RequireRoles`, region scoping, `isGlobalAdmin`), `OrgManagerGuard`, `DeptChatAccessGuard`, conversation `roleOf()`. Role staleness is a client/UX problem, not a server authz hole — except where a guard was never mounted (RS-01).
- **The upgrade transition UX works end-to-end:** provider signup re-pulls `/auth/me` and re-shells correctly; agent onboarding polls `agents.status` with AppState re-checks.
- **Department-channel membership rekey seam is properly engineered** (server never holds the master key; add/remove intents; deterministic convergent rekey planners in messenger-core) — the model RS-02 should copy.
- **Users-page role/KYC filter literals now match the DB** (`individual`/`approved` — fixed earlier today).

## 5. Recommended fix order

1. **RS-01** mount the CPO session gate + revoke JTIs in `setMemberStatus` (small, closes live access-control hole).
2. **RS-04** terminate/reject flips `users.role` back to `individual` (+ revoke sessions, + `user.role.change` audit action).
3. **RS-03 + RS-17/18/19** make the tier lifecycle real: apply `@RequireTier('pro')` to the Pro endpoints it was written for; drop the `pro_active_until IS NOT NULL` exemption from the sweep (decide the one NULL-expiry Pro row first); add a daily Stripe reconciliation (or include `past_due`-beyond-grace rows in the sweep); mobile treats `pro_active_until < now` as Lite locally.
4. **RS-05** mint messenger tickets from a fresh DB read; revoke JTIs on role flips.
5. **RS-11** add `user.role.change` / `member.role` / `admin.role` audit actions everywhere a role is written.
6. **RS-06/07** mobile: generalize `recheckMembership` to all shells on AppState resume; clear `pendingProvider` on failure/logout.
7. **RS-02/08** conversation rekey-on-remove + signed admin promotion — **needs architecture sign-off first** (group-key stop-condition).
8. **RS-12/13/14** migration: tighten `users_role_check` to the 3 live values (after deciding the Ops-1 row), constrain or rename `cpo_pool.role`, purge E2E fixtures.

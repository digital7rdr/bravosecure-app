## 1. Executive summary

**Verdict: the security core remains production-grade; availability/durability and the newest notification+call surfaces are not yet.** Across 11 completed areas the fleet confirmed **35 findings: 1 P0, 16 P1, 17 P2, 1 P3**, and correctly **refuted 6** false positives. **No E2EE or security-contract violation was found** — sealed-sender cert verification stayed unconditional, AAD binding held, the relay never saw plaintext or keys, no plaintext/key material reached logs or notifications, and the vault MFA gate stayed unbypassable everywhere read.

The **one P0 is a regression introduced by the 2026-07-06 M-14 fix**: the coalesced status-flush transaction and the receive transaction now run on the **same single SQLCipher connection under two independent mutexes**, so under the exact reconnect-drain + markRead burst M-14 optimized for, a receive `BEGIN IMMEDIATE` can land inside an open flush transaction, throw, and be **acked `discarded` — permanently destroying a committed inbound message** (unrecoverable for groups). This must be fixed before any production rollout.

The **16 P1s** cluster into five themes:

- **Durability / message-loss edges (send):** M-15 is only _partially_ fixed — the cert fetch (incl. its 30 s negative-cache throw) still runs _before_ the optimistic append, and the retry chip deletes the failed bubble before re-sending, so typed text can still be lost permanently; an offline cold boot **bricks the whole messenger for the process lifetime** (un-caught `publishOwnBundle` reject is cached); disappearing-message TTL is **silently ignored for all media**.
- **Group roster split-brain:** mobile remove/add/leave on server-backed (UUID) groups never writes the server roster, so the Home sync **resurrects removed members** (who then still receive fan-out media keys + download grants) and **drops added members**; and the plaintext-interop receive branch **skips the membership gate**, letting any authenticated non-member inject rendered messages into a group thread.
- **Notification correctness (the freshest code):** missed-call taps open a **ghost incoming CallScreen** for a dead call; a group message from a DM-contact still produces a **misattributed 1:1 banner + a duplicate group banner** and bypasses group mute (the exact bug N-11/N-14 claimed fixed); inline **Reply silently discards** the typed reply once the posting process has died.
- **Privacy enforcement gaps:** the client `blockedPeers` cache is **device-global, not owner-scoped** — on a multi-account device it silently destroys the _wrong_ user's messages; and **block is not enforced on calls** at all — a blocked user can still ring the blocker's lock screen, live or killed.
- **Transport / server call-signalling:** no single-flight guard on `open()`/`forceReconnect()` lets concurrent reopens **double-connect the same device**, and the server's same-device supersession then triggers a **spurious full sign-out**; the recovery handback sends `socket.id` instead of the private recovery `pid`, so the 2-minute missed-frame replay **never fires**; and 78edfd4's N-02 call-teardown shipped **mirror-image Redis-key bugs** — a decline clears the wrong device (phantom missed-call + ghost re-ring) and a caller-hangup-to-offline-device **orphans the missed marker** so the reconnect "Missed call" never lands (the headline B-53 fix is still broken for the common case).

The **17 P2s** are DoS-hardening gaps (vault/SFU/TURN/sfu.ring are the only endpoints with no per-user throttle — the "global" throttler is inert without an `APP_GUARD`), SFU worker-death room leaks, several "fixed" durability items that are still memory-only (read-receipt queue), and block-enforcement holes on reactions/group renders.

Separately, the tester reported **5 live-device call bugs** (§7) — an incoming-call notification that opens the app instead of the call screen, an **ongoing-call notification tap that disconnects the call**, and a broken connected-state cluster (wrong duration format, dead timer, un-updated status). These are not among the 35 static findings; preliminary anchors are given and a 2-device trace is queued.

## 3. 🔴 P0 — fix before any production rollout

### P0-1 · M-14 coalesced-flush races the receive txn on one SQLCipher connection → committed inbound message acked `discarded` and destroyed

- **Anchor:** `src/modules/messenger/store/sqlMessageStore.ts:327` (flush txn) vs `src/modules/messenger/runtime/receiveTransaction.ts:83` (receive txn); ack at `productionRuntime.ts:5189` / `:7265`
- **Regression source:** the 2026-07-06 M-14 fix (pre-fix status updates were autocommit single statements; the fix routed them through `upsertCoalesced` → a multi-statement `BEGIN…COMMIT`).
- **Scenario:** user opens a chat while `drainRelay` delivers a reconnect backlog. `markRead` bulk-flips ~30 bubbles → the write-through queues 30 coalesced updates → 50 ms later the flush opens `BEGIN` + 30 upserts + `COMMIT` (tens of ms of open txn). The next envelope's `runWithRatchetTxn` `BEGIN IMMEDIATE` executes **inside that window** → _"cannot start a transaction within a transaction"_ → `handleIncoming` throws before decrypt → `handledOk=false` → `relay.ack(envelopeId, 'discarded')` → **the relay deletes the envelope**; the sender sees "undelivered", the recipient never renders it. **Groups are unrecoverable** (no B-46 resend); 1:1 gets at most one sender-side auto-resend. The reverse interleaving silently drops the whole status batch (ticks revert after restart).
- **Why it's real:** two independent transaction mutexes (`SqlMessageStore.txMutex` and the module-level `receiveTransaction.txnChain`) both drive explicit multi-statement transactions on the **same** op-sqlite connection (`ownStore.getDb()`); neither checks the other. `sqlCipherStore.ts:160-167` already proves this failure mode on this stack and guards _its_ writes via `isInsideRatchetTxn()` — `upsertBatch` has no such guard. The catch-all then classifies **any** local throw as terminal and acks destruction.
- **Fix:** funnel `upsertBatch` through the `receiveTransaction` `txnChain` (single shared per-connection mutex), **and** reclassify transient local SQL errors (nested-txn / `SQLITE_BUSY` / disk pressure) as **leave-on-relay** instead of ack-`discarded`. The second half is the systemic fix — a local error should never cause the relay to delete a message it still holds.
- **Verifiers:** refute / reproduce / impact all CONFIRMED P0.

## 4. 🟠 P1 register (16)

### Durability & message-loss (send)

**P1-1 · M-15 incomplete: cert fetch still runs before the optimistic append; retry chip can destroy the failed message** — `productionRuntime.ts:2090`
The M-15 fix moved X3DH/seal/wrap below the append but left `await certCache.get()` (2090) above **both** appends (group 2230, 1:1 2502). When the network drops with the cert past its 10-min margin, `certCache.get()` rejects and arms a 30 s negative cache, so every send in that window insta-rejects _before_ the bubble exists — composer already cleared (`ChatScreen.tsx:601`), no bubble, no outbox row, no retry chip: **typed text gone**. Worse, tapping retry on a durable `failed` bubble runs `removeMessage` _first_ (`ChatScreen.tsx:675`) then `sendText`, which rejects at the cert fetch — **the persisted message is deleted, unrecoverable**. _Fix:_ append the optimistic bubble before _any_ await; in `retrySend` append the new `sending` bubble before removing the old one.

**P1-2 · Offline/failed cold boot permanently poisons the messenger** — `runtime.ts:472`
`buildProductionRuntime` hits `await publishOwnBundle(...)` (`productionRuntime.ts:497`) with **no try/catch**; `keys.uploadBundle` uses bare `fetch` and throws offline, so the build rejects _before_ SQLCipher history hydration and `setReady`. `runtime.ts:469-474` caches the **rejected** promise and never clears it on failure. Result: cold-start offline (or on a transient auth-service 5xx) → messenger shows zero history and every send is impossible **for the entire process lifetime**, even after connectivity returns, until force-kill. _Fix:_ make the boot bundle upload best-effort (retry on reconnect) and/or clear `runtimePromise` when `buildRuntime` rejects.

**P1-3 · Disappearing-message TTL silently ignored for ALL media** — `ChatScreen.tsx:895`
Text sends pass `ttlSeconds: ttlSec ?? convTtl` (621-625) but `sendPickedMedia` — the single funnel for camera, gallery, documents, and voice notes — calls `rt.sendMedia(...)` with **no `ttlSeconds`**. The runtime plumbing exists (`productionRuntime.ts:2778-2781`) but always receives `undefined`, so `expiresAtSec` is never stamped: images/video/voice notes/files in a disappearing conversation **never expire** on either side, no relay expiry, the R2 ciphertext + both local copies persist indefinitely. A silent failure of an advertised privacy feature. _Fix:_ pass `ttlSeconds: ttlSec ?? convTtl ?? undefined` (and the `isGroup` hint) in `sendPickedMedia`.

### Group roster split-brain & integrity

**P1-4 · Plaintext group-text fall-through bypasses the membership gate — any authenticated non-member can inject rendered messages into any group thread** — `productionRuntime.ts:6284`
A user with a valid sender cert (including a member removed via remove+rekey) seals a pairwise envelope carrying `group:{groupId:G, kind:'text'}` with a plaintext (non-`GroupCiphertext`) body. `parseGroupMessage` returns `malformed`, so the runtime's legacy branch renders it — builds `legacyMsg` with `conversation_id=G` and `appendMessage`s it into the group thread (auto-creating the group row). The `isGroupMember` drop at `:6639` only guards the master-key-decrypted path and **never runs for parse failures**. Outsiders or ousted members can plant spam/phishing into private ops-room threads on every member's device, persisted to SQLCipher. _Fix:_ apply the membership gate to the legacy/malformed branch (drop when local `GroupState` exists and the sender isn't a member; require the conversation row to list the sender otherwise).

**P1-5 · M-09 incomplete for server-backed groups: mobile remove/leave never updates the server roster → `/conversations/mine` resurrects the removed member into fan-out** — `ChatInfoScreen.tsx:354`
`removeGroupMember` does correct crypto (remove@E, rekey@E+1, shrinks `convo.participants`) but never calls `conversationApi.removeMember`, so the server `conversation_members` row survives. The next `MessengerHome` mount overwrites `participants` with the server list, re-adding the removed user. Every subsequent send fans out to them — text stays safe (new key) but `sealPayload` carries `attachment{keyB64,ivB64}` at the pairwise layer their intact Signal session decrypts, and `registerGrants(objectKey, participants)` **grants them the server download**. A removed insider reads all post-removal media, defeating remove+rekey. Same cause for a leaver. _Fix:_ route UUID-group remove/leave through `conversationApi.removeMember` (RS-02 intents reconcile), or stop the Home sync overwriting participants for a locally-crypto-owned group.

**P1-6 · Mobile add-member on a server-backed group is undone by server sync** — `NewChatScreen.tsx:193`
`runtime.addGroupMember` does the full add+rekey and adds the member to `convo.participants`, but never writes `conversation_members`. On every member's next Home mount the sync overwrites participants with the stale server list, **silently excluding the new member from future sends**; and on the _new_ member's device the UUID group row is absent from `/conversations/mine` so the prune loop deletes the thread. Net: the new member holds a valid key and can post but never receives replies and loses the thread — add-member silently fails for exactly the conversations RS-02 exists for. _Fix:_ route add-to-group through `conversationApi.addMember`, or exempt locally-crypto-owned groups from the participants overwrite/prune.

### Notification correctness (78edfd4 — freshest code)

**P1-7 · Missed-call notification tap navigates to a ghost incoming CallScreen** — `fcmBootstrap.ts:709`
After a caller rings a killed/backgrounded device and hangs up, N-02 posts a "Missed call" banner with `data:{kind:'missed-call', callId}`. The only tap handler falls into the CALL branch, opens a full-screen incoming-call UI (_Answering…_, Answer/Decline) for a dead call with `conversationId:'direct:'`, `remoteUserId:undefined`, `incomingSdp:undefined`. Answer can never connect (no offer exists). Fires on warm tap, cold-launch tap, and after **every** missed call — a golden-path interaction shipped broken by the same commit that made missed-call banners common. _Fix:_ branch on `data.kind==='missed-call'` before the callId fall-through → deep-link to the caller's thread / call log; include `fromUserId` in the payload.

**P1-8 · N-11/N-14 incomplete: group message from a DM-contact still gets a misattributed 1:1 banner + duplicate group banner; mute inversion survives** — `fcmBootstrap.ts:1199`
App backgrounded >1 min (Doze dropped the WS — the normal state), a group sender who also has a DM with the user posts in the group. The dup-banner skip only fires when **no** direct thread resolves; under sealed sender a group wake carries only `senderUserId`, which resolves to the DM id, so `skip` stays false and a banner keyed/deep-linked to the **1:1** is posted, then `pullEnvelopes()` decrypts the group message and the store notifier posts a **second**, correctly group-keyed banner (both alert). If the user muted the **group**, the misattributed "Alice" banner still rings; if they muted **Alice's DM**, the group wake is suppressed as muted. The audit shipped this as fixed; only the no-DM case is fixed. _Fix:_ when the notifier is running, skip the FCM banner regardless of `convId` resolution (the store notifier draws the correct banner after the pull).

**P1-9 · Inline Reply / Mark-as-read are dead once the posting process dies — typed reply silently discarded** — `callNotification.ts:429`
`backgroundMessageNotifier` posts message banners with `actions:true` (Reply + Mark-read), but the rich reply handler is registered only inside `startFcmBootstrap` (after login-mount). When Android LMK kills the process overnight (routine), a notification action relaunches **headless JS** where only the slim bundle-entry handler exists (`index.js:50`) — it handles `callId` only, so a `reply-<convId>` `ACTION_PRESS` **does nothing**: the typed text is dropped, the thread isn't marked read, the RemoteInput spinner hangs. The user believes they replied; the peer receives nothing. _Fix:_ persist `reply-<convId>` input to a durable pending-replies queue the runtime drains on next boot, or at minimum surface "Open app to send".

### Privacy enforcement

**P1-10 · `blockedPeers` cache is device-global (not owner-scoped) and only server-synced on a settings visit — cross-account block bleed silently destroys the wrong user's messages** — `blockedPeers.ts:20`
One global AsyncStorage key `messenger.blockedPeers.v1`, no `ownerUserId`, never reset on owner switch. On a multi-account device (supported via `vaultByOwner`), account B inherits account A's block set: every 1:1 message X sends B is decrypted, marked seen, acked `delivered`, and **dropped — permanent silent loss with no UI trace**. Conversely, when B opens Messenger Settings, `setBlockedPeers` replaces the set with B's server list, **silently unenforcing A's blocks**. After reinstall the cache is empty and blocks are unenforced until the user visits the blocked-list screen. _Fix:_ key storage per owner (like `restoreTombstones`), reset on owner switch, fetch `GET /users/blocked` during runtime build.

**P1-11 · Block (M-07) not enforced on calls — a blocked user can still ring the blocker's lock screen, live and killed** — `messenger.gateway.ts:1097`
`handleCallOffer` performs **no privacy check** — it forwards the offer to the victim's live socket, queues the pending offer, and fires `sendVoipWake` unconditionally, so the victim's phone shows a full-screen incoming-call ring even when killed. `fcmHeadless` voip-wake has no block check either; the client's `isPeerBlocked` is consulted only on the inbound _text_ path. The wake budget allows ~6 rings/min per pair indefinitely, so the harassment channel the block was meant to close stays fully open on the most intrusive surface. `handleSfuRing` (group ring) is equally ungated. Unlike sealed-sender messages, `call.offer` exposes both parties to the server, so a server-side gate is feasible. _Fix:_ silent-drop in `handleCallOffer` (mirroring the typing no-oracle pattern) when `isBlockedEither`; filter blocked targets in `handleSfuRing`.

### Transport & server call-signalling

**P1-12 · No single-flight guard on `TransportClient.open()`/`forceReconnect()` — concurrent reopens create two live sockets; same-device supersession then triggers a spurious full sign-out** — `packages/messenger-core/src/transport/client.ts:454`
Two reconnect triggers within `open()`'s async window (the 5 s send-ack watchdog racing an AppState-`active` reconnect, both suspended at `await getToken()`) each create a socket; the first becomes an orphan with live listeners. Both connect; the gateway evicts the first for the same `(userId, deviceId)` with `superseded`+disconnect, and `productionRuntime.ts:962-978` responds by calling `signOut()` — **the user is dumped to the login screen** though the "other device" was this device's own duplicate socket. Secondary: brief double frame dispatch (double call ring) and a leaked authenticated socket. _Fix:_ single-flight `open()` (coalesce on an in-flight promise + connect-generation counter), or have the server suppress client-visible `superseded` when the superseding connection shares the device + boot nonce.

**P1-13 · Recovery handback sends `socket.id` instead of the socket.io recovery `pid` — `connectionStateRecovery` never fires on rebuilt sockets, and the stale `auth.pid` breaks stock in-process recovery** — `packages/messenger-core/src/transport/client.ts:538`
socket.io-client 4.8.3 has no public `pid` (it's the private `_pid`), so `sock.pid` is always `undefined` and `socket.id` is captured/persisted/handed back. The server keys recovery by the private pid, so `restoreSession` always misses and every rebuild mints a fresh session — the **2-minute missed-frame replay never happens** (mid-blip `call.offer`/`answer`/`ice`, typing, presence, mission frames), the exact class Fix #17/#18 and the custom adapter were built to close. Worse, the client's CONNECT builder `Object.assign({pid:this._pid,...}, auth)` lets our wrong `auth.pid` **override** the correct internal `_pid`, permanently breaking in-process auto-reconnect recovery after the first rebuild. Cost with zero benefit: one AsyncStorage write per inbound frame persisting unusable offsets. _Fix:_ capture `(socket as any)._pid` (never fall back to `socket.id`) and only inject `auth.pid/offset` when a real recovery pid exists.

**P1-14 · N-02 decline path clears the wrong device's Redis keys → phantom "Missed call" + ghost re-ring after a callee declines a 1:1 call** — `messenger.gateway.ts:1223`
The queued offer/marker/index are always keyed on the **callee** (`handleCallOffer` uses `data.to`). On a callee **decline**, the hangup's `data.to` is the **caller**, so `clearPendingCallArtifacts` hits the caller's non-existent keys and the callee's own 6 h missed-marker + index entry **survive**. On the callee's next reconnect, `deliverPendingCallOffer` reads the surviving marker → emits `call.missed` → a "Missed call" bubble + notification for a call the callee **explicitly declined**; if the reconnect is within 45 s the offer payload is still fresh → a full **ghost incoming ring** for the already-declined call. _Fix:_ target the ringing callee (`pre.callee.*` / `ctx.claims.sub`) for the queued artifacts, not `data.to`.

**P1-15 · N-02 regression: hangup SREMs the pending-offer index even when keeping the missed-marker → the reconnect `call.missed` drain never fires** — `messenger.gateway.ts:1155`
When a caller hangs up (or the 45 s ring-timeout fires) against an offline/killed callee, `handleCallHangup` keeps the marker (`keepMarker:callerGaveUp`) but `clearPendingCallArtifacts` **unconditionally** SREMs the callId from `pending-call-offer-idx`. The reconnect drain enumerates **only** that index, so the surviving 6 h marker is unreachable and `call.missed` is never emitted; the parallel cancel push has a 60 s FCM TTL and is discarded for any device offline >60 s. Net: the callee comes back online hours later with **no missed-call record anywhere** — the exact B-53 complaint this fix shipped to close. The marker path only works when the caller's socket dies without sending hangup (crash), i.e. almost never. _(Same root as the P2 caller-gives-up finding at `:1155`.)_ _Fix:_ move the `srem` inside the `if (!opts.keepMarker)` block; consider raising the cancel-push FCM TTL.

**P1-16 · Dedup claim not released on non-queue-full put failure → retried send returns fake success and the message is silently lost for 30 days** — `apps/messenger-service/src/relay/envelope.service.ts:194`
`claimClientMsgId` (SET NX) succeeds, then `store.put` fails with any error **other than** `PendingQueueFullError` (Redis OOM under `noeviction`, a connection drop between the two awaits, a failover mid-submit). The bare `throw e` leaves the claim in place. The client outbox retries → hits the still-claimed dedup key → `submitEnvelope` returns the cached `{envelopeId, retractToken}` as **success** with no envelope in Redis (and no archive row). The sender's UI paints "sent"; the recipient never receives it; the dedup key blocks every re-attempt for 30 days. _Fix:_ release the dedup claim in a catch-all (any put failure), keeping the queue-full special case for the 429 mapping — or claim only after a successful put.

### 12.2 backup / restore

**P1‑B‑1 · Sealed-archive replay is unresumable and fail-silent — the restore-incomplete marker is cleared BEFORE the archive drain, and any drain error is swallowed while the success overlay shows** — `src/screens/messenger/BackupRestoreScreen.tsx:446`
`restoreAllMessages` clears the H-2 restore-incomplete marker on its own completion (`restoreMessages.ts:643-646`), but the `sealed_envelope_archive` replay — which recovers every message received while the mirror was locked (the reinstall window) — runs **after** that and replays up to ~500k envelopes with per-envelope libsignal decrypts. Killed mid-drain (Doze/OOM) → next boot sees `isRestoreIncomplete=false` → RESUME path → **the archive is never re-pulled; those messages are permanently absent while the user was told restore completed.** Even without a crash, a transient network error mid-drain is caught at `:446`, logged, and falls through to the SUCCESS overlay (`:489`). No per-page archive cursor persistence (`cursorMs`/`cursorId` are loop locals) so partial progress is always discarded. _Fix:_ clear `clearRestoreIncomplete` (or add an `archive-replay-incomplete` marker) **after** the archive drain; persist the archive tuple cursor per owner (like `restoreResume.ts`); surface a retry state on drain failure instead of the success overlay.

**P1‑P‑1 · `TOTP_ENCRYPTION_KEY` defaults to a hardcoded, publicly-known key — 2FA seed at-rest encryption is worthless if the env var is unset in prod** — `apps/auth-service/src/config/configuration.ts:41`

## 6. Conventions used for these fixes

- **Fail closed by default.** Every previously-fail-open path now fails closed; rollout escape hatches are env-var-driven, default off, and emit `console.warn` at module load if set.
- **Honest UX over polished lies.** Where a feature wasn't actually working, the UI now says so out loud — pickers still open, alerts say "your file was not saved or sent", encryption badges describe the real state. No fake-failed bubbles, no green "ACTIVE" for things that aren't active.
- **Small commits, each independently revertable.** One audit code per commit; the chain can be cherry-picked or bisected.
- **No behaviour change without a test failing first.** New crypto behaviour (safety number, `requireAad`) ships with regression tests in the same commit. Tests that asserted the old fail-open behaviour are flipped to assert the fail-closed reason and explicitly cite the audit code.
- **Mirror the package.** When a fix touches `packages/messenger-core/src/crypto/*`, the mobile mirror in `src/modules/messenger/crypto/*` is updated in the same commit. The duplicate-module problem (C1 in the groups audit) is on the pending list; until it lands, both copies move together.

## 7. How to add the next fix to this tracker

1. Pick an item from §4 (or a new finding from a follow-up audit).
2. Write the failing test first when practical; mark the audit code in the test name.
3. Implement the smallest patch that closes the finding.
4. Commit with `fix(<area>): <one-line> (audit <code>)` and a `Co-Authored-By` trailer if collaborating with Claude.
5. Move the row from §4 into §1 with the actual commit hash, the file anchors that changed, and the new test count if any.
6. If the fix re-enables a switch from §5, remove that row from §5.

The goal is for §4 to shrink monotonically.

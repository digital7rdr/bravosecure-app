# Virtual Bodyguard (VBG) — Full Feature Audit

**Date:** 2026-07-05
**Auditor:** Claude (full read of mobile `src/screens/vbg/*`, `src/modules/vbg/*`, backend `apps/auth-service/src/vbg/*`, API client, navigation wiring, DB verification against live Supabase)
**Score: 64 / 100** — excellent engineering quality and graceful degradation, but the core "bodyguard" promise (autonomous monitoring + escalation) has product-level gaps that make it a _dashboard_, not yet a _guarantee_.

### Backend (`apps/auth-service/src/vbg/`, NestJS)

- `vbg.controller.ts` — JWT-guarded + per-user throttled; every endpoint scoped to `user.sub` (no foreign-id acceptance). Panic 3/min, telemetry 25/min, heartbeat 12/min.
- `vbg.service.ts` (997 lines) — monitoring enroll/heartbeat/biometric check-in, encrypted telemetry ingest (Redis stream + durable last-fix), panic (reuses `SosService.raise`), SRA scoring, threat blend, key points, favorites.
- Intel sources: `gdelt.service.ts` (serialized ≥5.2s spacing, 15-min cache), `newsdata.service.ts` (optional, key-gated), `googlenews.service.ts` (RSS, free), all classified by shared `threatClassify.ts`.
- `geocode.service.ts` — Mapbox reverse geocode, ~1 km grid cache, 1 h TTL.
- `geofence.service.ts` — PostGIS polygon containment, transition-only breach fan-out (Kafka + ops feed + WS + SMS).
- `telemetryCrypto.ts` — AES-256-GCM, base64(iv‖ct‖tag), per-device server-issued key.

### Database (verified live in Supabase, 2026-07-05)

All six tables exist: `vbg_monitoring` (10 cols), `vbg_device_keys` (4), `vbg_telemetry_last` (6), `vbg_sra_snapshots` (8), `vbg_geofences` (7), `vbg_favorites` (7).

### Data flow (SRA / threats)

GPS fix → Mapbox reverse geocode → place hierarchy → **parallel, time-budgeted blend** (GDELT 7 s / NewsData 6 s / Google News RSS 6 s) → dedupe by headline → severity classification → score = `min(100, critical×8 + caution×3)` → persisted snapshot (`vbg_sra_snapshots`).

## 2. What is genuinely good

- **Graceful degradation everywhere.** Every upstream (GDELT throttle, Overpass mirror down, NewsData unset, geocode miss) degrades to a usable state; per-source time budgets (`withTimeout`) keep the SRA inside the client's 15 s axios timeout. Null-Island (0,0) fixes are rejected instead of scored.
- **Correct concurrency discipline.** GDELT calls serialized behind a promise chain with 5.2 s spacing; favorites reconcile under a per-user `pg_advisory_xact_lock` with a guard that rejects an all-invalid list rather than silently wiping contacts.
- **Escalations reuse the proven SOS path** (`SosService.raise`) — panic/missed-scan/geofence all land in the same Ops-Room feed + Kafka audit + SMS; breach alerts fire only on **state transitions**, so a stationary principal doesn't spam ops.
- **Sensible security posture.** All endpoints JWT + per-user throttle; SQL fully parameterized; the one string-interpolated SQL literal (EWKT polygon) re-validates every coordinate as finite + range-clamped first; telemetry is AES-256-GCM with per-device keys held in Keychain (`WHEN_UNLOCKED_THIS_DEVICE_ONLY`); plaintext coordinates are never logged.
- **Classifier pragmatics.** Benign-context suppression ("stock market rally", "fire sale", cricket results) plus word-start boundaries ("fire" ≠ "ceasefire", "raid" ≠ "afraid") — with unit tests.
- **Solid service-level test coverage.** 23 tests on `VbgService` covering escalation timing, radius/window threading, favorites edge cases (silent-wipe guard, advisory lock), telemetry decrypt failures, 3rd-fail escalation; plus geofence, classifier, Google News parser, footer route-table, and stale-GPS-coords tests.
- **Offline emergency directory** — ~180 countries bundled, device-locale pinning, universal 112 fallback; zero network dependency at the worst moment.

## 3. Findings

### HIGH — the "bodyguard" promise is not yet autonomous

| ID      | Finding                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **H-1** | **No server-side watchdog for missed scans.** Escalation logic runs only _inside_ `heartbeat()`/`biometricCheckin()` — i.e. only when the client phones home. A dead battery, confiscated phone, or force-killed app **never escalates**: the `overdue` flag is computed on read, but nothing polls it. The SRA screen explicitly promises "If 3 scans are missed → Ops Room alert → call → Live Action Protocol". Needs a scheduled job (cron/BullMQ) scanning `vbg_monitoring` for `active` rows past `interval_min × 3` and raising SOS server-side. |
| **H-2** | **The client never performs the scans it enrolls for.** `vbgApi.heartbeat` and `vbgApi.biometricCheckin` exist in `src/services/api.ts` but **no screen calls them**. "Enable Face Scan Monitoring" enrolls (and stores the telemetry key) and then… nothing: no scheduled prompt, no notification, no MLKit face-capture flow (which exists elsewhere in the app for Dept Chat check-in and could be reused). Combined with H-1, the entire missed-scan escalation chain is currently unreachable end-to-end.                                          |
| **H-3** | **Monitoring stops when the screen closes.** The 3 s encrypted-telemetry loop lives in a `useEffect` on `VBGHomeScreen` — foreground-only, screen-only. No foreground service / background location mode exists (the Mapbox audit already flagged the unused background-location manifest declarations). Ops sees a live dot only while the principal stares at the dashboard.                                                                                                                                                                          |
| **H-4** | **Escalation SMS goes to the principal, not the Next of Kin.** `contactPhone()` resolves the _principal's own_ `users.phone_e164`; panic texts "PANIC raised…" to the person in duress. The server-backed `vbg_favorites` (built precisely for this) are never used in any fan-out (panic, biometric escalation, geofence breach).                                                                                                                                                                                                                      |

### MEDIUM

| ID       | Finding                                                                                                                                                                                                                                                                                                                                                                                   |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **M-1**  | **Overpass key-points query only matches `node[...]`.** Most large hospitals, police HQs and embassies are mapped as **ways/relations** in OSM, so they're invisible to `/vbg/keypoints`. Use `nwr[...]` + `out center` (same query shape; `el.center.lat/lon` fallback). This materially under-reports safe havens — the feature's whole purpose.                                        |
| **M-2**  | **"Location History — last 24 hours" is actually ~last 25 minutes.** The Redis stream is capped `MAXLEN ~500`; at the 3 s cadence that's ≈25 min of fixes (TTL 24 h notwithstanding). UI copy overpromises; either raise MAXLEN/downsample or say "recent trail". On any Redis error the fallback silently returns a single last fix presented as "history".                              |
| **M-3**  | **Geofencing has no UI.** Full CRUD API + PostGIS eval + client API methods exist, but no screen lists/creates/deletes zones — and the Home card shows a hardcoded **"Geofence Active"** badge to every user, including those with zero zones. A false safety signal in a security product.                                                                                               |
| **M-4**  | **`ingestTelemetry` throws bare `Error`** for un-enrolled keys / bad ciphertext → HTTP 500s instead of 400/401. Pollutes error monitoring and makes client-side handling indistinguishable from real server faults.                                                                                                                                                                       |
| **M-5**  | **No AAD binding / replay protection on telemetry.** The GCM envelope binds nothing (no userId/deviceId/timestamp AAD); `recordedAt` is trusted from the payload. An attacker holding a valid JWT could replay captured blobs to fake a stationary track. Cheap fix: AAD = `userId‖deviceId`, reject `recordedAt` older than N minutes.                                                   |
| **M-6**  | **~5 DB queries per user per 3 s tick.** Each ingest does a last-fix upsert + geofence eval (2 PostGIS containment queries + monitoring SELECT + UPDATE) even for users with **zero zones**. Cache "has zones" per user (or check `total=0` first) and skip the eval; consider batching last-fix writes.                                                                                  |
| **M-7**  | **Unbounded in-memory caches.** GDELT / NewsData / GoogleNews / Geocode all use `Map` caches keyed by region/place/window with TTL checks on read but **no eviction** — a long-lived process accumulates entries forever (GeoRisk lets users query arbitrary places). Add a max-size sweep.                                                                                               |
| **M-8**  | **Risk score is media-volume-sensitive, not normalized.** `critical×8 + caution×3` scales with news-coverage density: a big English-speaking metro trends HIGH while a genuinely dangerous low-coverage area reads LOW (compounded by `language=en` on NewsData and English-only classifier keywords — non-Latin local press is invisible). Document the bias in ops training at minimum. |
| **M-9**  | **The Home "Live Location" map is decorative.** `TacticalMap` is a fake SVG street grid with a hardcoded locator at (46 %, 56 %) and a hardcoded route. It looks like live cartography on a screen whose other numbers are real. Replace with the real `VbgKeyPointsMap` mini-view (the map work shipped alongside this audit makes it a drop-in), or visually mark it as schematic.      |
| **M-10** | **SRA snapshots persist an incomplete record.** Only lat/lng/score/risks/recommendations are stored — not `region`, `level`, `summary`, or the counts shown to the principal. If ops ever needs "what was the user told?" the record can't answer.                                                                                                                                        |

### LOW

| ID  | Finding                                                                                                                                                                                                                                                            |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| L-1 | `vbgKeyPointsMapHtml` defines `.kp .tip` label CSS that is never rendered — key-point pins have no on-map labels (names only in the list below the map).                                                                                                           |
| L-2 | Date parsers (`parseGdeltDate`, `parseNewsDataDate`, `parsePubDate`) fall back to `now()` for missing/malformed dates — fabricated recency then survives `withinWindow` filtering and severity sorting. Prefer dropping or flagging undated items.                 |
| L-3 | `useVbgLocation` is one-shot (`maximumAge: 60 s`) with no refresh on screen focus — a principal who moves and returns to a mounted screen keeps the stale fix.                                                                                                     |
| L-4 | GeoRisk `run()`'s deferred key-points continuation calls `setKeypoints` without the `aliveRef` guard used elsewhere in the same file (setState-after-unmount warning).                                                                                             |
| L-5 | Emergency-screen country pinning matches the Mapbox **English country name** against the bundled directory names — fragile for names with variants ("Côte d'Ivoire" vs "Ivory Coast"); ISO short_code (already available in the geocode context) would be robust.  |
| L-6 | `VBGSRAScreen` `enrolled` state is local-only — it resets on remount even when the server row is active (`monitoringStatus` is never read), so the button re-reads "ENABLE…" after re-visiting.                                                                    |
| L-7 | `vbg_device_keys.key_b64` is plaintext at rest in Postgres. Acceptable for the documented threat model (server is the trusted decryptor; this is transport encryption, not E2EE) — but worth a `// Why:` note and a KMS/pgcrypto consideration for prod hardening. |

## 4. Security review summary

- **AuthZ:** ✅ all endpoints derive identity from JWT claims; no user-supplied IDs.
- **Injection:** ✅ parameterized SQL; EWKT literal is numerically re-validated; ops-console map labels HTML-escaped; agent-tracker callsign via `textContent`; VBG map never injects server text into HTML.
- **Crypto:** ✅ AES-256-GCM done correctly (fresh 12-byte IV per message, tag verified, key length asserted); isolated from libsignal per architecture constraints. ⚠️ M-5 (no AAD/replay), L-7 (key at rest).
- **Secrets/logging:** ✅ no plaintext coordinates or keys logged; decrypt failures rejected "without leaking".
- **Rate limiting:** ✅ sensible per-endpoint budgets; escalations themselves SOS-throttled.

## 5. Test coverage assessment

- Backend: **good** at the service layer (`vbg.service.spec.ts` 23 tests, `geofence.service.spec.ts`, `googlenews.service.spec.ts`, `threatClassify.spec.ts`). Gap: no controller-level test (guards/throttle wiring), no test for the Overpass node-vs-way shape (would have caught M-1), no telemetryCrypto round-trip test on the client.
- Mobile: footer route-table and GeoRisk stale-coords logic are unit-tested (both extracted into pure modules — nice pattern). The screens' data flows are untested (acceptable for this codebase's conventions).

## 6. Priority recommendations

1. **Close the escalation loop (H-1 + H-2):** a server cron over `vbg_monitoring` overdue rows, plus a client scheduled scan prompt (local notification → MLKit face capture → `biometricCheckin`). This is the single change that turns VBG from a dashboard into a bodyguard.
2. **Wire Next-of-Kin into the fan-outs (H-4):** panic / biometric escalation / geofence breach should SMS the favorites (they're already server-side and E.164-normalized).
3. **Fix key-points coverage (M-1):** `node[...]` → `nwr[...]` + `out center` — a ~6-line change with large real-world impact.
4. **Ship or hide geofencing (M-3):** add a minimal zones screen, or remove the hardcoded "Geofence Active" badge.
5. Background telemetry (H-3) is a bigger lift (FGS + iOS background mode) — schedule it; the manifest groundwork exists.
6. Batch the M/L hygiene fixes (error codes, cache eviction, honest copy) into a maintenance pass.
